3-1.プルリクエスト

所属しているチームでコードレビューしてますか？
コードレビューする文化は根付いていますか？

開発の始めには"必ずコードレビューする"とルールを決めていても、「忙しくてレビューできない」「レビュー対象・変更箇所がわからない」などの理由で形骸化していくことは、よくある話である

このように、コードレビューの文化を根付かせることは難しい

しかし、Gitのプルリクエストを使うことがコードレビュー文化を根付かせる一助となる

プルリクエストとは

プルリクエストとは、開発者のローカルリポジトリでの変更を他の開発者に通知し、リモートリポジトリへの取り込みを要求する機能

  • プルリクエストは、Git自体ではなく、GitHubが提供する機能
  • GitLabでは、マージリクエストと呼ばれる同様の機能がある
プルリクエストが提供する機能
  • 変更内容をレビュー担当者に通知する
  • ソースコードの変更箇所・差分を可視化する
  • ソースコードに関するコミュニケーションの場を提供する

プルリクエストのメリット

レビュー・マージ作業をタスク化して管理し、やりとりを記録できる

  • 作成されたプルリクエストは一覧で見ることができる
  • プルリクエスト上でコメントをやりとりして議論できる
  • プッシュされたコミットは、自動的にプルリクエスト上に反映される

プルリクエストを利用した開発プロセス

プルリクエストを利用した開発プロセスは、以下の流れとなる

  1. [ 開発者 ] 作業対象のソースを clone または pull する
  2. [ 開発者 ] 作業用のブランチを作成する
  3. [ 開発者 ] ソースコードを追加・修正する
  4. [ 開発者 ] 作業が完了したら add, commit, push する
  5. [ 開発者 ] プルリクエストを作成する
  6. [ レビュー担当者 ] 通知されたプルリクエストから変更を確認しレビューする
  7. [ レビュー担当者 ] レビュー結果を判断し、必要に応じて開発者にフィードバックする
  8. [ レビュー担当者 ] レビューの結果、問題がない場合はマージする
  9. [ レビュー担当者 ] レビューの結果、対応自体が不要となるなど、プルリクエスト自体が必要ない場合はクローズする

上記の3〜7の工程を、必要な分だけ繰り返す
その結果、最終的にマージされるソースコードの品質が向上される
